# Git Workflow 미션

---

## 1. 미션 소개

Git은 단순한 버전 관리 도구가 아니라, **팀 협업의 핵심 인프라**입니다. 아무리 코딩을 잘해도 Git을 제대로 다루지 못하면 팀 프로젝트에서 병목이 됩니다.

이 미션에서는 **3~5인 팀**으로 실제 협업 상황을 시뮬레이션합니다. 같은 파일을 동시에 수정하여 충돌을 경험하고, **PR**로 코드 리뷰를 주고받으며, **브랜치 전략**을 적용합니다. 단순히 명령어를 외우는 것이 아니라, *왜 이 워크플로우가 필요한지*를 팀원들과 함께 체득합니다.

이 미션의 경험은 실무에서 바로 적용됩니다. 특히 **PR 기반 협업**과 **코드 리뷰 문화**는 현업에서 가장 중요한 소프트 스킬 중 하나입니다.

## 3. 과제 목표

이 과제를 마친 후, 학습자는 아래를 **스스로 설명**할 수 있어야 한다.

1. **브랜치**가 내부적으로 어떻게 동작하는지(커밋을 가리키는 포인터), 그리고 **왜 브랜치를 나눠서** 작업하는지
2. **GitHub Flow**가 무엇이고, 왜 팀 협업에서 이런 전략이 필요한지
3. **PR(Pull Request)** 의 목적과 **코드 리뷰**의 가치
4. **충돌**이 왜 발생하고, 충돌 마커(`<<<<<<<`, `=======`, `>>>>>>>`)가 무엇을 의미하는지
5. **`reset`**, **`revert`**, **`stash`** 의 차이와 각각 언제 사용하는지
6. 팀 협업 중 문제가 생겼을 때(충돌·실수 커밋 등) **기록을 남기고** 재현 가능한 방식으로 해결 과정을 설명할 수 있는지

## 4. 기능 요구 사항

다음 요구사항을 **모두** 만족해야 한다.

### 4.1 팀 구성 및 저장소 준비

| 항목 | 요구 사항 |
|------|-----------|
| **팀 규모** | **3~5인** |
| **저장소** | 아래 **옵션 A** 또는 **B** |

| 옵션 | 설명 |
|------|------|
| **A (권장)** | GitHub **Organization** 저장소 생성 |
| **B (대체)** | 개인 저장소 생성 후 **Collaborator** 초대 |

**기본 폴더 · 파일 구조**

| 경로 | 용도 |
|------|------|
| `README.md` | 프로젝트 소개, 실행 방법 등 |
| `docs/` | 문서 |
| `src/` | 코드 또는 예시 파일 |
| `team/` | 팀원별 소개 문서 (선택 시) |

### 4.2 Branch Protection Rule (`main`)

**`main` 브랜치**에 아래를 설정한다.

| 규칙 | 요구 사항 |
|------|-----------|
| **직접 push** | 금지 |
| **병합** | **PR을 통해서만** 허용 |
| **승인** | 최소 **1명 approve** 필요 |

### 4.3 브랜치 전략 (GitHub Flow)

| 브랜치 | 역할 |
|--------|------|
| **`main`** | 항상 배포 가능한 상태(팀 기준에서 "깨지지 않는 상태") |
| **`feature/*`** | 작업 단위 브랜치 |

| 항목 | 요구 사항 |
|------|-----------|
| **네이밍** | 브랜치 네이밍 규칙을 정하고 `docs/CONTRIBUTING.md`에 문서화 |
| **선택 이유** | GitHub Flow를 선택한 이유를 **3줄 이내**로 `README.md` 또는 `docs/CONTRIBUTING.md`에 기록 |

### 4.4 이슈 기반 작업 및 PR 연동 (필수)

| 항목 | 요구 사항 |
|------|-----------|
| **이슈** | 각 작업은 **Issue**로 생성하고 PR과 연동 |
| **PR 본문** | `Closes #이슈번호` 또는 `Fixes #...` 포함으로 이슈 추적 가능 |
| **`SUBMISSION.md`** | 팀원별 "내가 만든 Issue/PR"을 빠르게 확인 가능 |

### 4.5 커밋 메시지 컨벤션

| 항목 | 요구 사항 |
|------|-----------|
| **문서화** | 팀 규칙을 `docs/CONTRIBUTING.md`에 명시 |
| **권장 형식 (예시)** | `feat: subject`, `fix: subject`, `docs: subject`, `refactor: subject` 등 |

**"의미없는 메시지"로 간주되는 경우** (아래 중 하나라도 해당)

| 유형 | 예시 |
|------|------|
| 변경 대상을 유추할 수 없는 단어만 있는 경우 | `update`, `fix`, `temp`, `wip`, `final` 등 |
| 무엇을/왜 바꿨는지 드러나지 않는 경우 | `bug fix`, `edit file` 등 (구체 대상·효과 없음) |

### 4.6 PR 기반 협업 (팀원별 기여도)

| 항목 | 요구 사항 |
|------|-----------|
| **병합 경로** | 모든 `feature` 브랜치는 **PR로** `main`에 병합 |

**팀원별 최소 기준** (전원 충족)

| 기준 | 최소 횟수 |
|------|-----------|
| PR 생성 및 병합 | **2개** |
| 코드 리뷰 작성 (본인 PR 제외) | **2개** |
| 리뷰 반영 경험 (본인 PR) | **1회 이상** (커밋·수정·답글로 증빙) |

**PR 본문 최소 포함 항목** (형식은 자유)

| 항목 | 내용 |
|------|------|
| **What** | 변경 사항 |
| **Why** | 변경 이유 |
| **How** | 테스트·검증 방법 |

### 4.7 코드 리뷰 최소 품질 기준

| 항목 | 요구 사항 |
|------|-----------|
| **코멘트** | "LGTM/좋아요"만이 아닌 **실질 코멘트 1개 이상** — 특정 라인·파일 근거의 질문, 대안 제안, 리스크 지적, 개선 제안 등 |
| **상호작용** | 리뷰어와 작성자 간 **최소 1회** 답글·수정 반영이 기록으로 남아야 함 |

### 4.8 충돌 해결 실습 (비자명 충돌 포함)

| 범위 | 요구 사항 |
|------|-----------|
| **팀 전체** | **2회 이상** 충돌 해결 기록 (커밋·PR·문서) |
| **비자명 충돌** | **최소 1회** (아래 중 하나 이상) |

| 유형 | 설명 |
|------|------|
| **같은 파일, 인접 hunk** | 같은 파일의 같은 hunk(인접 라인)를 서로 다르게 수정 |
| **이동·삭제 vs 내용 수정** | 한쪽은 파일 이동·이름 변경(또는 삭제), 다른 한쪽은 내용 수정 (머지·리베이스 시 이슈 유발) |

충돌 해결 과정은 **`docs/conflict-resolution.md`** 에 기록한다 (템플릿 준수 권장).

### 4.9 Git 트러블슈팅 실습 (팀 전체 4종 + 팀원 참여)

**팀 전체가 아래 4가지 시나리오를 모두 수행하고 문서로 남긴다.**

| 시나리오 | 설명 |
|----------|------|
| `git commit --amend` | 최근 커밋 메시지 수정 |
| `git reset --soft HEAD~1` | 로컬 커밋 취소 + 변경 유지 |
| `git revert` | 원격에 push된 커밋 취소 |
| `git stash` / `git stash pop` | 작업 보관 후 전환 |

| 항목 | 요구 사항 |
|------|-----------|
| **팀원별** | 각 팀원은 **최소 1개** 시나리오 해결 기록 작성에 참여 (이름·역할이 문서에 드러나야 함) |
| **기록 위치** | **`docs/troubleshooting-log.md`** |

### 4.10 협업 가이드 문서 (`docs/CONTRIBUTING.md`)

팀원이 분담하여 작성. **최소 포함 내용:**

| 항목 |
|------|
| 브랜치 네이밍 규칙 |
| 커밋 메시지 컨벤션 |
| PR 작성 규칙 (본문에 포함할 항목) |
| 코드 리뷰 규칙 (리뷰 최소 품질 기준 포함) |
| 충돌 발생 시 기본 대응 흐름 (누가 / 어떻게 / 어디에 기록) |

### 4.11 간단한 결과물 (택 1) + 팀원별 최소 기준

아래 중 **하나**를 선택한다 (복잡한 기능 구현이 목적이 아님).

| 선택 | 내용 |
|------|------|
| **(A) 유틸 함수 모음** | 팀원별 함수 **1개 이상** + 간단 사용 예시 (`README` 또는 docstring) |
| **(B) 팀 소개** | `team/`에 팀원별 소개 파일 + `README`에서 링크·목록 제공 |
| **(C) 학습 노트** | 팀원별 노트 **1개 이상** + `README`에서 목차 제공 |

| 항목 | 요구 사항 |
|------|-----------|
| **팀원별** | 선택한 결과물에 대해 각 팀원 **최소 1건의 기여 커밋** |

## 5. 보너스 과제 (선택)

| 항목 | 내용 |
|------|------|
| **히스토리 정리** | `git rebase -i`로 개인 `feature` 브랜치의 커밋을 squash/reword하여 정리. 정리 전·후 히스토리를 간단히 문서화 |
| **CODEOWNERS · 리뷰어 자동화** | `.github/CODEOWNERS`를 추가해 파일·폴더별 책임 리뷰어 지정 |
